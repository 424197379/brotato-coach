# 验收标准

## 商店建议

- 给定角色、材料、当前装备和四个商店候选，输出有序购买方案。
- 购买总价不得超过可用材料。
- 能识别立即合成、不同武器计数、套装跨阈值和锁定价值。
- 每个动作至少包含一条正向依据和一条主要代价。
- 缺少价格时不得声称精确剩余材料。

## 属性诊断

- 输出当前最大的三个缺口，按严重程度排序。
- 每个缺口包含当前值、目标区间、时间窗口和影响。
- 能识别角色特有的属性换算和武器双重缩放。
- 不得仅依据固定全局阈值忽略角色和构筑。

## 未来规划

- 同时输出当前波次加 3 和加 5 的目标。
- 目标必须覆盖输出、生存、回复、移动、经济和武器结构。
- 当经济不足时提供保守方案。
- 当输入来自模组内容时标注规则覆盖度和置信度。

## 结算复盘

- 输出最早断档波次，而不只分析死亡波次。
- 至少比较相邻三个波次的属性、经济和伤害趋势。
- 区分直接死因与更早的运营根因。
- 输出三条可在下一局执行的改进规则。

## 游戏内体验

- 分析仅由按钮触发。
- 商店界面提供“教练建议”按钮，并基于当前 `ShopItem.value`、`locked` 和 `item_data` 生成建议。
- 暂停菜单提供“分析当前局”按钮。
- 结算界面提供“复盘本局”按钮。
- 分析面板可关闭、可滚动，不遮挡必要的继续或结算操作。
- 三个入口复用同一个结果面板，且面板支持键鼠和手柄焦点。
- 无记录、字段缺失或规则异常时不崩溃。
- 离线状态下功能完整可用。
- 游戏内分析只执行 GDScript 和本地规则数据，不调用外部 Node、AI 或网络。
- 分析期间不修改玩家材料、属性、装备或存档。
- Mod 打包为本地 ZIP；若 Brotato 正在运行，不覆盖已加载 Mod、不改 profile、不重启游戏。

## 性能与回归

- 核心规则对标准样本的单次执行低于 100 ms。
- 波次快照低于 10 ms。
- 相同输入、规则版本和配置产生相同 JSON 报告。
- 每个已支持角色至少有三个快照样本和一个完整跑局样本。

## P1 垂直切片

- Case 002 商店报告必须包含新武士刀和伤疤 `buy_now`，其余商店候选 `lock`。
- Case 002 必须输出属性缺口和 `+3`/`+5` 波规划。
- Case 001 复盘必须输出更早防御/移动欠账、诅咒未闭环、输出增速落后和黑暗祭坛最终触发点。
- 外部 CLI 必须输出稳定 JSON 和 Markdown，但不得替代游戏内按钮闭环。
- 构建产物必须说明待游戏退出后的安全安装/注册步骤。
